シングルページアプリ【SPA】＝ 単一HTMLファイル

  • アプリケーション全体が1つのindex.htmlから始まる
  • ページ遷移時も同じHTMLファイル内で完結
  • JavaScriptがDOMを直接操作してコンテンツを切り替える
  • URLが変わる場合もあるが、実際にはページリロードしない


<!DOCTYPE html>
<html>
<head>
    <title>React App</title>
</head>
<body>
    <div id="root"></div> <!-- ここにReactがコンテンツを描画 -->
    <script src="bundle.js"></script> <!-- アプリケーションのコード -->
</body>
</html>
SPAは「シングルページアプリケーション」の略りゃくです。
普通ふつうのWebサイトは複数ふくすうのHTMLファイルがありますが、
SPAは1つのHTMLファイルだけを使つかいます。
ページを移動いどうしてもHTMLファイルは変かわりません。
JavaScriptが画面がめんの内容ないようを変更へんこう します。

状態 ＝ 再レンダーを引き起こす変数


import React, { useState } from 'react';

function LikeButton() {
  // [現在の値, セッター関数] = useState(初期値)
  const [likes, setLikes] = useState(0);

  return (
    <div>
      <button onClick={() => setLikes((prevLikes) => prevLikes + 1)}>
        ♡ {likes}
      </button>
    </div>
  );
}

動作の流れ

  1. useState(0) → 初期値0でいいね数を作成
  2. ボタンクリック → setLikes(likes + 1)
  3. 状態更新 → コンポーネント再レンダリング
  4. 新しい値でUI更新

重要：状態が変わるたびに関数コンポーネント全体が再実行される
この 例（れい）では、likes（らいくす）という 名前（なまえ）で 数（かず）を 覚（おぼ）えています。
ボタンを 押（お）すと、setLikes（せっとらいくす）で 数（かず）が ひとつ 増（ふ）えます。
そうすると、画面（がめん）の 数字（すうじ）も 変（か）わります。

制御されたコンポーネント ＝ 状態 ＋ イベントハンドラー


基本構造

// 3つの要素が必須
const [value, setValue] = useState('');        // 1. 状態

return (
  <input 
    value={value}                              // 2. input要素
    onChange={(e) => setValue(e.target.value)} // 3. ハンドラー関数
  />
);

重要なポイント

  • valueは必ずstate由来
  • onChangeでstateを更新
  • 状態変更 → 再レンダリング
  • Reactが完全に制御
・「Controlled」では、Reactが扱います
・「Uncontrolled」では、Reactが状態を知らない、DOMに任せます
制御せいぎょされたコンポーネントは、Reactが入力にゅうりょく値あたい を状態じょうたい 経由けいゆで一元管理いちげんかんりするしくみ。
ユーザーが入力にゅうりょくするとonChangeイベントが発火はっかして、状態 じょうたいが更新 こうしん される。
inputの表示ひょうじ値あたいは常つね にStateに依存いぞん。
この構造こうぞうにより、検証けんしょうや制御せいぎょ が容易よういになる。

なぜ純粋コンポーネントが必要なのか？

コンポーネント(props, state) → JSX

Reactの要求:

  • 同じ入力 → 同じ出力
  • レンダー中に副作用を起こさない

理由: レンダー中、Reactは勝手に

  • コンポーネントを複数回実行
  • 途中でキャンセル・やり直し
  • 最適化のため結果を破棄

副作用 ＝ 外部世界との相互作用
レンダー中に副作用を実行すると → いつ何回実行されるかわからない

管理された副作用

ユーザーアクション時 ✅
function handleClick() {
  fetch('/api/data'); // 予測可能
}
コンポーネント状態変化時 ✅
useEffect(() => {
  fetch('/api/data'); // 安全なタイミング
}, []);
レンダー中 ❌
function Component() {
  fetch('/api/data'); // 同じAPIを何回も叩く
  document.title = "バカ"; // タイトルがチカチカする
  ...
}

エフェクト ＝ 副作用を「いつ実行するか」で制御する仕組み


import { useEffect } from "react";

function Component() {
    function cleanup() { // クリーンアップ関数：リソースの解放
        console.log("アンマウント時や次のエフェクト実行前");
    }

    function setup() { // エフェクト関数：副作用を実行
        console.log("DOM更新後に実行される");
        return cleanup;
    }

    useEffect(setup, [/* 依存配列 */]);

    return <div>...</div>;
}
エフェクト関数
副作用を実行する関数
クリーンアップ関数（オプション）
リソース解放
依存配列（オプション）
実行制御
  • 依存配列なし → 毎回実行
  • 空の依存配列 [] → 初回のみ
  • 値を指定 [userId] → 値変更時のみ
セットアップ関数かんすうからクリーンアップ関数かんすう を返かえ すことができます。
クリーンアップ関数かんすうでリソースを片付かたづけます。
クリーンアップはコンポーネントが消きえる前まえ に実行じっこうされます。

コンポーネントのライフサイクル

マウント

  1. コンポーネント初期化
  2. 初回レンダリング
  3. DOM更新
  4. useEffect実行
画面に追加された時
  • 初回ページ読み込み
  • 親コンポーネントが初回作成
  • 条件付きレンダリング
    {show && <Component />} // show → true

更新

  1. stateやpropsの変更検知
  2. 再レンダリング
  3. DOM差分更新
  4. 前のエフェクトクリーンアップ
  5. 新しいエフェクト実行
propsやstateが変化した時
  • setState呼び出し
  • 親からpropsが変更
  • useEffect依存配列の値変更

アンマウント

  1. 全エフェクトクリーンアップ
  2. DOMから削除
  3. メモリ解放
画面から削除された時
  • 親コンポーネントが削除
  • ページを閉じる/離脱
  • 条件付きレンダリング
    {show && <Component />} // show → false

組み込みの React フック

Reactはフックを順序で管理する

function MyComponent() {
    const [name, setName] = useState("");     // フック #1
    const [age, setAge] = useState(0);        // フック #2
    useEffect(() => { ... }, []);             // フック #3
    const [email, setEmail] = useState("");   // フック #4

Reactが混乱 → エラーまたは予期しない動作

フックのルール

要するに：毎回同じ順序でフックを呼び出す

ルール1：トップレベルでのみ呼び出し

  • ループ、条件分岐、ネストした関数内で
    呼び出してはいけない
  • コンポーネントやフックの最上位で呼び出す

ルール2：React関数からのみ呼び出し

  • Reactコンポーネントから呼び出す
  • またはuseで始まる関数から呼び出す

フックのルール違反例

条件分岐内でのフック

function Component({ condition }) {
  if (condition) {
    const [state, setState] = useState(0); // NG!
    
    useEffect(() => {
      // ...
    }, []);
  }
  
  return <div>...</div>;
}

正しい書き方

function Component({ condition }) {
  const [state, setState] = useState(0); // OK!
  
  useEffect(() => {
    if (condition) {
      // 条件分岐はフック内部で
    }
  }, [condition]);
  
  return <div>...</div>;
}
フックの実行順序が変わると、Reactの内部状態管理が破綻します
ルールを守まもらないとエラーになったりバグが起おきます。

まとめ

  • Reactは フックを呼び出し順序で管理している
  • 毎回同じ順序で呼ぶ必要がある（ルール）
  • ルールを破ると予期しない動作やエラーが発生

カスタムフック

コンポーネント間でのロジック共有

カスタムフック

既存のフックを組み合わせて作る独自のフック
// 1. 名前は「use」で始める（必須）
function useToggle(initialValue = false) { 
    // 2. 組み込みフックを最上位で呼び出す
    const [value, setValue] = useState(initialValue); 
    
    // 3. 必要なロジックを書く
    const toggle = () => setValue(prev => !prev);
    
    // 4. 必要なロジックを書く
    return [value, toggle];
}
  • useStateを内部で使っている → これはフック
  • 複数のコンポーネントで再利用できる
  • 同じフックのルールが適用される
➡
使用例：通常のフックと同じように使う
function Modal() {
    // カスタムフックを呼び出す
    const [isOpen, toggleOpen] = useToggle(false); 
    
    return (
        <>
            <button onClick={toggleOpen}>
                {isOpen ? '閉じる' : '開く'}
            </button>
            {isOpen && <div>モーダル内容</div>}
        </>
    );
}

// 他のコンポーネントでも同じロジックを再利用可能
function Sidebar() {
    const [isVisible, toggleVisible] = useToggle(true);
    // ...
}

何がフックを「フック」にするのか

フック = 他のフック（組み込みのReact機能）を呼び出す関数

実際のフック

useStateを呼び出している → フック

function useCounter(initialValue = 0) {
    const [count, setCount] = useState(initialValue);
    
    const increment = () => setCount(prev => prev + 1);
    
    return [count, increment];
}

フックではない普通の関数

他のフックを呼び出していない → ただの関数

function formatPrice(price) {
    return `¥${price.toLocaleString()}`;
}

フックは他のフックを呼び出せる

カスタムフック同士も組み合わせ可能
// 基本的なフック
function useToggle(initialValue = false) {
    const [value, setValue] = useState(initialValue);
    const toggle = () => setValue(prev => !prev);
    return [value, toggle];
}

// 他のカスタムフックを使うカスタムフック
function useModal() {
    const [isOpen, toggleOpen] = useToggle(false);  // カスタムフック呼び出し
    const [data, setData] = useState(null);         // 組み込みフック呼び出し
    
    const openModal = (modalData) => {
        setData(modalData);
        toggleOpen();
    };
    
    return { isOpen, data, openModal, closeModal: toggleOpen };
}

カスタムフック

共通状態ロジックを抽出
function useApi(url) {
    const [data, setData] = useState(null);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState(null);
    
    useEffect(() => {
        setLoading(true);
        fetch(url)
            .then(res => res.json())
            .then(setData)
            .catch(setError)
            .finally(() => setLoading(false));
    }, [url]);
    
    return { data, loading, error };
}
➡
使用例：
function UserProfile({ userId }) {
    const { data, loading, error } = useApi(`/api/users/${userId}`);
    
    if (loading) return <div>ユーザーを読み込み中...</div>;
    if (error) return <div>エラー</div>;
    
    return <div>{data.name}</div>;
}

function PostList() {
    const { data, loading, error } = useApi('/api/posts');
    
    if (loading) return <div>投稿を読み込み中...</div>;
    if (error) return <div>エラー</div>;
    
    return data.map(post => <div>{post.title}</div>);
}
