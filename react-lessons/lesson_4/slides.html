コンポーネント ＝ 関数

// コンポーネント定義（大文字で開始）
function UserCard({ name, email, children }) {
  return (
    <div className="card">
      <h3>{name}</h3>
      <p>{email}</p>
      {children}
    </div>
  );
}

// 使用
<UserCard name="Kurotsuchi Mayuri" email="m.kurotsuchi@seireitei.co.jp">
  <button>編集</button>
</UserCard>
props = 関数の引数
children = タグの中身
Reactのコンポーネントは関数かんすうとして書か きます。
名前なまえは必かならず大文字おおもじ から始はじめます。
この関数かんすうは、propsプロップスというデータを受う け取とって、JSXを返かえします。

状態 ＝ 再レンダーを引き起こす変数


import React, { useState } from 'react';

function LikeButton() {
  // [現在の値, セッター関数] = useState(初期値)
  const [likes, setLikes] = useState(0);

  return (
    <div>
      <button onClick={() => setLikes(likes + 1)}>
        ♡ {likes}
      </button>
    </div>
  );
}

動作の流れ

  1. useState(0) → 初期値0でいいね数を作成
  2. ボタンクリック → setLikes(likes + 1)
  3. 状態更新 → コンポーネント再レンダリング
  4. 新しい値でUI更新

重要：状態が変わるたびに関数コンポーネント全体が再実行される
この 例（れい）では、likes（らいくす）という 名前（なまえ）で 数（かず）を 覚（おぼ）えています。
ボタンを 押（お）すと、setLikes（せっとらいくす）で 数（かず）が ひとつ 増（ふ）えます。
そうすると、画面（がめん）の 数字（すうじ）も 変（か）わります。

制御されたコンポーネント ＝ 状態 ＋ イベントハンドラー


基本構造

// 3つの要素が必須
const [value, setValue] = useState('');        // 1. 状態

return (
  <input 
    value={value}                              // 2. input要素
    onChange={(e) => setValue(e.target.value)} // 3. ハンドラー関数
  />
);

重要なポイント

  • valueは必ずstate由来
  • onChangeでstateを更新
  • 状態変更 → 再レンダリング
  • Reactが完全に制御
・「Controlled」では、Reactが扱います
・「Uncontrolled」では、Reactが状態を知らない、DOMに任せます
制御せいぎょされたコンポーネントは、Reactが入力にゅうりょく値あたい を状態じょうたい 経由けいゆで一元管理いちげんかんりするしくみ。
ユーザーが入力にゅうりょくするとonChangeイベントが発火はっかして、状態 じょうたいが更新 こうしん される。
inputの表示ひょうじ値あたいは常つね にStateに依存いぞん。
この構造こうぞうにより、検証けんしょうや制御せいぎょ が容易よういになる。

シングルページアプリ【SPA】＝ 単一HTMLファイル

  • アプリケーション全体が1つのindex.htmlから始まる
  • ページ遷移時も同じHTMLファイル内で完結
  • JavaScriptがDOMを直接操作してコンテンツを切り替える
  • URLが変わる場合もあるが、実際にはページリロードしない


<!DOCTYPE html>
<html>
<head>
    <title>React App</title>
</head>
<body>
    <div id="root"></div> <!-- ここにReactがコンテンツを描画 -->
    <script src="bundle.js"></script> <!-- アプリケーションのコード -->
</body>
</html>
SPAは「シングルページアプリケーション」の略りゃくです。
普通ふつうのWebサイトは複数ふくすうのHTMLファイルがありますが、
SPAは1つのHTMLファイルだけを使つかいます。
ページを移動いどうしてもHTMLファイルは変かわりません。
JavaScriptが画面がめんの内容ないようを変更へんこう します。

なぜ純粋コンポーネントが必要なのか？

コンポーネント(props, state) → JSX

Reactの要求:

  • 同じ入力 → 同じ出力
  • 外の世界に直接触らない

理由: レンダー中、Reactは勝手に

  • コンポーネントを複数回実行
  • 途中でキャンセル・やり直し
  • 最適化のため結果を破棄
  • 開発環境では2回実行

副作用 ＝ 外部世界との相互作用
外の世界を直接触ると → 副作用がいつ何回実行されるかわからない

管理された副作用

ユーザーアクション時 ✅
function handleClick() {
  fetch('/api/data'); // 予測可能
}
コンポーネント状態変化時 ✅
useEffect(() => {
  fetch('/api/data'); // 安全なタイミング
}, []);
レンダー中 ❌
function Component() {
  fetch('/api/data'); // 同じAPIを何回も叩く
  document.title = "hello"; // タイトルがチカチカする
  ...
}

エフェクト ＝ 副作用を「いつ実行するか」で制御する仕組み


import { useEffect } from "react";

function Component() {
    function cleanup() { // クリーンアップ関数：リソースの解放
        console.log("アンマウント時や次のエフェクト実行前");
    }

    function setup() { // エフェクト関数：副作用を実行
        console.log("DOM更新後に実行される");
        return cleanup;
    }

    useEffect(setup, [/* 依存配列 */]);

    return <div>...</div>;
}
エフェクト関数
副作用を実行する関数
クリーンアップ関数（オプション）
リソース解放
依存配列（オプション）
実行制御
  • 依存配列なし → 毎回実行
  • 空の依存配列 [] → 初回のみ
  • 値を指定 [userId] → 値変更時のみ
セットアップ関数かんすうからクリーンアップ関数かんすう を返かえ すことができます。
クリーンアップ関数かんすうでリソースを片付かたづけます。
クリーンアップはコンポーネントが消きえる前まえ に実行じっこうされます。

コンポーネントのライフサイクル

マウント

  1. コンポーネント初期化
  2. 初回レンダリング
  3. DOM更新
  4. useEffect実行
いつ起きる？
  • 初回ページ読み込み
  • 親コンポーネントが初回作成
  • 条件付きレンダリング
    {show && <Component />} // show → true

更新

  1. stateやpropsの変更検知
  2. 再レンダリング
  3. DOM差分更新
  4. 前のエフェクトクリーンアップ
  5. 新しいエフェクト実行
いつ起きる？
  • setState呼び出し
  • 親からpropsが変更
  • useEffect依存配列の値変更

アンマウント

  1. 全エフェクトクリーンアップ
  2. DOMから削除
  3. メモリ解放
いつ起きる？
  • 親コンポーネントが削除
  • ページを閉じる/離脱
  • 条件付きレンダリング
    {show && <Component />} // show → false

カスタムフック

コンポーネント間でのロジック共有

フックのルール

フックを正しく使用するための重要な制約


ルール1：トップレベルでのみ呼び出し

  • ループ、条件分岐、ネストした関数内で呼び出してはいけない
  • コンポーネントの最上位で呼び出す
  • フックの実行順序を保証

ルール2：React関数からのみ呼び出し

  • Reactコンポーネントから呼び出す
  • カスタムフックから呼び出す
  • 通常のJavaScript関数からは呼び出してはいけない
フックは「use」で始はじまるReactの特別とくべつな関数かんすう です。
今いままで使つかっていたuseStateもuseEffectもフックです。
フックを使つかうときは必かならずルールを守まも ってください。

フックはReactコンポーネントの最上位さいじょういでのみ呼よび出だ し可能かのうです。

フックのルール違反例

条件分岐内でのフック

function Component({ condition }) {
  if (condition) {
    const [state, setState] = useState(0); // NG!
    
    useEffect(() => {
      // ...
    }, []);
  }
  
  return <div>...</div>;
}

正しい書き方

function Component({ condition }) {
  const [state, setState] = useState(0); // OK!
  
  useEffect(() => {
    if (condition) {
      // 条件分岐はフック内部で
    }
  }, [condition]);
  
  return <div>...</div>;
}

ループ内でのフック

function Component({ items }) {
  return items.map(item => {
    const [state, setState] = useState(0); // NG!

    return <div key={item.id}>{item.name}</div>;
  });
}
フックの実行順序が変わると、Reactの内部状態管理が破綻します
ルールを守まもらないとエラーになったりバグが起おきます。

まとめ

理解度チェック⑤【フックのルール】

このコードの問題は？

function Component({ showData }) {
  if (showData) {
    const [data, setData] = useState(null);
    
    useEffect(() => {
      fetchData().then(setData);
    }, []);
  }
  
  return <>...</>;
}
A) fetchDataが定義されていない
B) 条件分岐内でフックを呼んでいる
C) useEffectの依存配列が間違い
D) 特に問題なし
答え: B) フックは必ずトップレベルで呼び出す必要がある
